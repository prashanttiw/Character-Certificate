## 6. API surface & sample contracts

> **Note:** All endpoints should validate inputs, rate-limit key actions (OTP, login, verification), and return consistent error formats.

### Auth & registration

* `POST /api/auth/register/start` — body: `{ name, rollNo, email, mobile }` → sends registration OTP, does not create user yet.
* `POST /api/auth/register/verify` — body: `{ emailOrMobile, otp }` → on success creates user, returns JWT.
* `POST /api/auth/login` — body: `{ identifier (email|rollNo), password }` → returns JWT + refresh token.
* `POST /api/auth/send-otp` — reusable OTP endpoint for forgot-password or sensitive updates.
* `POST /api/auth/forgot-password/start` & `/verify` — two-step with OTP.

**Sample success response**:

```json
{ "ok": true, "message": "OTP sent", "next": "verify" }
```

### Certificate application

* `POST /api/certificates` — multipart form. Create or save draft (status `Draft`). Returns saved application id.
* `PUT /api/certificates/:id/submit` — marks `status=Submitted`. Triggers admin notification.
* `GET /api/student/dashboard` — returns latest application(s) and statuses for the logged-in student.
* `GET /api/certificates/:id` — get application + certificate metadata (only PDF URL if `Completed`).

### Admin

* `GET /api/admin/certificates` — list + filters (status, date, student)
* `POST /api/admin/certificates/:id/approve` — body: `{ comment? }` → generate PDF, set `Completed`, attach `certificateId`, send email.
* `POST /api/admin/certificates/:id/reject` — body: `{ comment }` → set `Rejected`, notify student.
* `POST /api/admin/certificates/:id/hold` — body: `{ comment, reason }` → set `OnHold`.

### Public verification

* `GET /api/verify?certificateId=CC-2025-0001` — returns `{ valid: true, name: "X", rollNo: "Y", issuedDate: "..." }` or `{ valid: false }`.

## 7. File uploads & handling

**Current:** implementation uses `uploadMiddleware` and stores files in `/uploads`.

**Production recommendations:**

* Use S3 or S3-compatible object storage, with strict bucket policies and versioning.
* Store file metadata in DB; keep original filename, content-type, size, checksum (SHA-256) to detect duplicates.- Validate file types (PDF/PNG/JPG only) and limit size (e.g., 5–10 MB).
* Scan files for malware / sanitize images (virus-scan/clamAV in pipeline) if possible.
* Use pre-signed URLs for direct client-to-S3 uploads to avoid overloading backend.

---

## 8. Certificate generation & digital signature

**Approach**:

* Keep a HTML/CSS certificate template so the certificate is deterministic and brand-consistent.
* Use Puppeteer to render the HTML to a print-ready PDF (supports fonts, exact layout, images).
* Embed unique `certificateId`, `issuedDate`, authority signature image, and a QR code linking to verification page.

**Digital / cryptographic signature options (prioritized):**

1. **MVP**: Visual signature image + seal (simple and widely accepted, but not cryptographically verifiable).
2. **Improved**: Sign the certificate PDF byte hash using a private key (RSA/ECDSA). Store signature metadata in DB and publish public key for verification.
3. **Advanced**: Anchor a certificate hash on a blockchain (e.g., Ethereum or a low-cost anchor) plus a revocation list for revoked certificates.

**Recommendation**: Implement option 1 initially for deliverability, then incrementally add option 2 for tamper-evidence.

## 9. Verification flow

1. Public page accepts `certificateId` (or scans QR).
2. Back-end finds the certificate and returns minimal verified attributes (name, rollNo, issuedDate, issuer).
3. Optionally, verify cryptographic signature (if implemented) and display verification status.

Security: avoid returning PII beyond what’s needed; rate-limit requests; log verification attempts.

## 10. OTP & sensitive update flows

**Current design:** two-step flows for registration and forgot password. OTPs are not persisted in DB (in-memory or transient store).

**Recommendations:**

* Move from in-memory to Redis for distributed systems and reliability.
* Store OTP hashed (e.g., HMAC) with TTL. Compare hashed version on verification.
* Limit OTP send attempts per phone/email per time window. Block suspicious activity.
* Log OTP attempts in ActivityLog.

## 11. Authentication & session management

* Passwords: bcrypt with appropriate salt rounds.
* JWT: sign with `process.env.JWT_SECRET`, include short-lived access token (e.g., 15m) + refresh token stored with rotation.
* Logout/invalidate tokens: maintain token blacklist or use rotating refresh tokens persisted in DB.
* Multi-factor: keep OTP-based login optional, especially for sensitive actions (profile change before certificate issuance).

## 12. Encryption notes (important, candid)

**Current implementation (per project context):** `encryptField` and `decryptField` utilities perform AES-256-CBC encryption derived from `process.env.SECRET_KEY` and use a fixed IV. All DB queries for encrypted fields use `encryptField` before querying.

**Reality check (tell it like it is):** Using a *fixed IV* with AES-CBC is insecure because the same plaintext will produce identical ciphertexts across records and leaks patterns. It also makes deterministic queries possible but weakens confidentiality and is vulnerable to certain ciphertext manipulation attacks.

**Actionable, non-sugarcoated recommendations:**

1. **Switch to random IV per encryption**: generate a secure random IV and store it concatenated with the ciphertext (e.g., base64(iv) + ":" + base64(ciphertext)). On decrypt, extract IV and decrypt.
2. **Use authenticated encryption**: prefer AES-256-GCM which provides integrity (auth tag). If you must use CBC, add HMAC for integrity.
3. **Indexing of encrypted fields**: If you need to query by email/mobile, maintain an additional deterministic index column by storing a keyed HMAC (e.g., HMAC-SHA256 using a separate KEY) that can be indexed — **do not** use raw ciphertext for DB queries.
4. **Key rotation**: implement key rotation strategy and key versioning metadata with each record.

Implementing these three changes will move your project from brittle to production-appropriate.

The First and Most Important Change: Separating Your Server from Your App
Right now, your server.js file is doing two jobs:

Configuring the Express application (setting up CORS, routes, etc.).

Starting the HTTP server to listen for requests.

Best Practice: We should split these into two files:

app.js: This file will set up and export our Express application. It will contain all the rules and routes. It's like the blueprint of our car.

server.js: This file's only job is to import the app, connect it to the internet (start the server), and connect it to the database. It's like turning the key in the ignition.

Why we do this: This separation is crucial for testing. Our tests can import the app directly to test our API routes without actually needing to start a live server and use a network connection. This makes our tests faster, more reliable, and is the professional standard.

2-> Step 2: Run the Redis Command
Once Docker Desktop is running, open your regular terminal (like Git Bash, Command Prompt, or PowerShell) and run this single command:

Bash

docker run -d -p 6379:6379 --name my-redis-server redis
What this command does:

docker run: The main command to run a container.

-d:  (Detached mode) This tells Docker to run the container in the background, so it doesn't take over your terminal.

-p 6379:6379: (Port mapping) This forwards traffic from port 6379 on your computer to port 6379 inside the container. Redis uses this port by default.

--name my-redis-server: This gives your container a memorable name.

redis: This is the name of the official Redis image on Docker Hub. Docker will automatically download it for you if you don't have it.
