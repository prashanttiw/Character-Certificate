------------------------------------------------------------
1. Project Summary
------------------------------------------------------------

The project currently has a working student-facing web application connected to a live Node.js + Express + MongoDB backend.

The strongest completed areas right now are:

- OTP-based student registration
- Student login with JWT
- Forgot-password flow with OTP
- Student dashboard connected to backend data
- Certificate application draft creation with file uploads
- Draft update flow
- Final submission of the latest draft for review
- Basic integration tests for the main student flow

The project is not yet complete as a full institution-ready certificate platform. Important features such as admin review panel, approval/rejection workflow UI, public certificate verification, PDF generation, and final certificate issuing are still not fully implemented in the active runtime path.

This means the system is best described as:

"A working student portal and backend foundation for a character certificate system, with core onboarding and application submission implemented, and administrative issuance workflows still pending."

2. Student fills name, roll number, email, mobile, and password

7. Known Issues, Risks, and Technical Debt
------------------------------------------------------------

This section is important because future planning should be based on the real gaps.

7.1 Encryption design needs improvement

Current implementation in `backend/utils/encryptionUtils.js` uses:

- AES-256-CBC
- derived key from `SECRET_KEY`
- fixed IV of all zeros

This is not production-grade encryption for sensitive user data.

Current effect:

- same plaintext gives same ciphertext
- pattern leakage is possible
- deterministic lookup works, but security is weaker than it should be

Recommended future fix:

- move to random IV per record
- prefer AES-256-GCM
- use separate indexed hash/HMAC field for lookup if deterministic email lookup is needed

7.2 OTP storage is inconsistent

Current state:

- registration OTP uses redis service with fallback
- forgot-password OTP uses in-memory store directly

Risk:

- behavior differs across flows
- in-memory OTP is not good for scaled or restarted environments

Recommended future fix:

- unify all OTP flows on Redis-backed TTL storage

7.3 Duplicate/legacy structure still exists

Current examples:

- `Student.js` re-export wrapper exists alongside `Student.model.js`
- inactive route files still remain in repo
- commented route mounting in `app.js`

Risk:

- confusion for future development
- accidental edits in unused files

Recommended future fix:

- cleanup and standardize module usage after next feature milestone

7.4 Upload storage is local only

Current state:

- files stored locally in uploads folder

Risk:

- not suitable for multi-server or production deployment
- files can be lost with server reset or deployment issues

Recommended future fix:

- move to S3 or equivalent object storage

7.5 Email delivery setup is basic

Current state:

- Gmail transporter
- skipped if credentials are missing

Future need:

- production-grade transactional provider
- better retry and logging strategy

16. Redis OTP Plan And Learning Guide
------------------------------------------------------------

Redis should be used as the central OTP store for this project.

16.1 Why Redis is the right choice

Problem with in-memory OTP:

- OTP disappears when server restarts
- not safe if multiple backend instances are running
- inconsistent between features

Why Redis works well:

- fast key-value store
- built-in expiry support with TTL
- good for temporary data
- ideal for OTP, sessions, counters, cooldowns, and rate limits

16.2 What data we should store in Redis

For each OTP flow, store:

- identifier such as email
- OTP or hashed OTP
- purpose such as `register`, `forgot-password`, `profile-edit`
- optional user payload for registration
- creation time
- expiry time handled by Redis TTL
- attempt count if we want brute-force protection

Recommended key format:

- `otp:register:<email>`
- `otp:forgot-password:<email>`
- `otp:profile-edit:<email>`

16.3 How OTP flow should work

Step 1: user requests OTP

- backend validates request
- backend generates 6-digit OTP
- backend stores OTP data in Redis with TTL like 10 minutes
- backend sends OTP through email

Step 2: user submits OTP

- backend reads Redis key
- backend checks if key exists
- backend compares submitted OTP
- if valid, backend completes action
- backend deletes key after successful verification

Step 3: optional security checks

- increment failed attempts
- block after too many failed attempts
- enforce resend cooldown

16.4 Recommended implementation changes in this project

Current target:

- keep `backend/services/redis.service.js`
- expand it into a generic OTP service
- stop using `backend/utils/otpStore.js` directly for forgot-password flow

Suggested service methods:

- `saveOtpData(purpose, identifier, otp, data)`
- `getOtpRecord(purpose, identifier)`
- `verifyOtpAndGetData(purpose, identifier, enteredOtp)`
- `clearOtpData(purpose, identifier)`
- `incrementOtpAttempts(purpose, identifier)`

Then update:

- registration controller to use this format
- forgot-password controller to use same Redis service
- future profile edit controller to use same Redis service

16.5 Better security version

Instead of storing plain OTP in Redis:

- hash OTP before saving
- compare hashed submitted OTP during verification

Why:

- even temporary data should follow least-exposure principles

Simple approach:

- generate OTP
- hash with SHA-256 or HMAC
- store hash in Redis
- email plain OTP to user
- hash entered OTP during verify and compare

16.6 Example development steps for Redis OTP feature

Step 1:

- refactor Redis service to support purpose-based keys

Step 2:

- move forgot-password OTP logic from `otpStore.js` into Redis service

Step 3:

- add TTL and delete-after-success verification in one place

Step 4:

- add attempt count and resend cooldown keys

Step 5:

- write integration tests for success, invalid OTP, expired OTP, and too many attempts

Step 6:

- remove old direct in-memory OTP usage except test fallback if needed

16.7 What you will learn by implementing Redis OTP

- how Redis differs from MongoDB
- when to use temporary storage instead of permanent storage
- how TTL works
- how to design verification flows
- how to secure OTP against brute force and replay
- how services abstract repeated backend logic
